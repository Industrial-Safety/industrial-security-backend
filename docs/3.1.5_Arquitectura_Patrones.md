# 3.1.5. Arquitectura de Desarrollo Impulsado por Patrones

La arquitectura de la plataforma **Industrial Safety Backend** se fundamenta en un enfoque de **Desarrollo Impulsado por Patrones** (Pattern-Driven Development), estructurada bajo un ecosistema distribuido de microservicios robusto, desacoplado y de alta cohesión. Cada microservicio está desarrollado con **Spring Boot (versión 4.0.6)** y **Java 25**, aplicando patrones de diseño consagrados para garantizar la flexibilidad, la mantenibilidad y la escalabilidad del sistema.

Para la fundamentación de esta sección, **se excluyen los componentes puros de infraestructura** (`api-gateway`, `config-server` y `discovery-server`), cuyo único propósito es el enrutamiento de red, la distribución de configuraciones y el registro/descubrimiento de servicios respectivamente. El análisis se centra exclusivamente en los **9 microservicios funcionales y de negocio** que componen la lógica de la aplicación:

1.  `user-service` (Gestión de usuarios, roles y autenticación con Keycloak).
2.  `safety-service` (Detección de EPP, cálculo de puntajes y gestión de incidentes).
3.  `course-service` (Administración de cursos, secciones e instructores).
4.  `exam-service` (Creación de exámenes, intentos de alumnos y emisión de certificados).
5.  `order-service` (Procesamiento de órdenes de compra, cupones y descuentos).
6.  `payment-service` (Integración de pasarela de pagos e historial transaccional).
7.  `chat-service` (Comunicación síncrona/asíncrona y foros de consulta).
8.  `notification-service` (Consumo de eventos y envío de alertas y correos).
9.  `purchase-service` (Inventariado de EPP, solicitudes de compra y actas de entrega).

## 1. Patrón Modelo-Vista-Controlador (MVC)

El patrón **Model-View-Controller (MVC)** es el núcleo estructural de cada uno de los microservicios funcionales. Al tratarse de una arquitectura orientada a APIs RESTful y microservicios, el patrón tradicional se adapta para desacoplar completamente la capa de presentación (Vista) del procesamiento lógico y el almacenamiento de datos.

```mermaid
graph TD
    subgraph Cliente
        App[Aplicación Frontend Web / Móvil]
    end
    subgraph Capa de Controladores REST (Controller)
        Ctrl[RestControllers]
    end
    subgraph Capa de Negocio y Modelo (Model)
        Svc[Services & Business Logic]
        Entity[Entities JPA / MongoDB Documents]
    end
    subgraph Capa de Datos (DAO / Repositorio)
        Repo[JPA / Mongo Repositories]
    end
    subgraph Almacenamiento Físico
        DB_SQL[(PostgreSQL)]
        DB_NoSQL[(MongoDB)]
    end

    App -->|Petición HTTP JSON| Ctrl
    Ctrl -->|Valida e invoca lógica| Svc
    Svc -->|Consulta / Persistencia| Repo
    Repo <-->|JPA / ORM| DB_SQL
    Repo <-->|Mongo Driver| DB_NoSQL
    Repo -->|Retorna Datos Físicos| Svc
    Svc -->|Mapea a DTO| Ctrl
    Ctrl -->|Retorna Payload JSON (Vista Desacoplada)| App
```

### Implementación en la Capa REST:
*   **Controladores (`controller`):** Clases anotadas con `@RestController` que actúan como puntos de entrada de tráfico HTTP. Gestionan el mapeo de recursos mediante anotaciones como `@GetMapping`, `@PostMapping` y `@PutMapping`. Su única responsabilidad es recibir solicitudes, realizar una validación sintáctica preliminar (mediante `@Valid`) y delegar la ejecución a la capa de servicios, retornando respuestas HTTP tipadas (`ResponseEntity`).
*   **Modelo (`model` / `entity` / `domain`):** Representa el estado y el comportamiento lógico del sistema. Está compuesto por:
    *   *Entidades Relacionales (JPA):* Clases en el paquete `model` (o `entity`/`domain`) anotadas con `@Entity` mapeadas a PostgreSQL (por ejemplo, `User.java` en `user-service` o `Incident.java` en `safety-service`).
    *   *Documentos No-Relacionales:* Clases anotadas con `@Document` mapeadas a MongoDB para gestionar datos no estructurados o de rápido crecimiento (por ejemplo, `Course.java` en `course-service` o `Conversation.java` en `chat-service`).
    *   *Servicios de Negocio:* Paquete `service` e `service.Impl` donde se ejecutan las reglas complejas y validaciones (por ejemplo, `IncidentServiceImpl.java` resolviendo la lógica de deducción de puntos al validar una infracción).
*   **Vista (Desacoplada):** No existe renderizado de plantillas HTML en el backend. La "Vista" se define contractualmente a través de payloads estructurados en formato **JSON**, los cuales se devuelven al cliente para que las aplicaciones Web (React/Next.js) o móviles los rendericen dinámicamente.

## 2. Objetos de Transferencia de Datos (DTO) y Mapeadores

La comunicación entre el cliente y el backend no expone directamente las entidades de base de datos. Para evitar problemas de sobreexposición (como sobreescritura maliciosa de campos o fuga de datos sensibles) y optimizar el ancho de banda, se implementa de manera estricta el patrón **Data Transfer Object (DTO)**.

*   **Clases DTO (`dto`):** Objetos de Java puros (POJOs o Java Records) que definen el contrato de entrada y salida de los endpoints. Por ejemplo, `UserRequest` define los datos necesarios para registrar un usuario, mientras que `UserResponse` expone datos procesados omitiendo campos de seguridad internos.
*   **Mapeadores Automáticos (`mapper`):** Para evitar la sobrecarga de código de conversión repetitivo (`getters` y `setters`), se utiliza la librería **MapStruct**. MapStruct genera implementaciones altamente eficientes en tiempo de compilación a través de interfaces anotadas con `@Mapper(componentModel = "spring")`. Por ejemplo, `UserMapper` traduce automáticamente entre `User`, `UserRequest` y `UserResponse` de forma segura y optimizada.

## 3. Data Access Object (DAO) / Patrón Repositorio

El acceso a la capa de persistencia física está completamente abstraído por el **Patrón Repositorio (Repository Pattern)**. Esto unifica y encapsula la forma en que los microservicios acceden a los datos, independientemente de si la base de datos subyacente es SQL o NoSQL.

*   **Abstracción Relacional (JPA):** Interfaces en el paquete `repository` que heredan de `JpaRepository<T, ID>`. Spring Data proporciona consultas CRUD, paginación y ordenamiento automáticos. Ejemplos de uso incluyen:
    ```java
    public interface UserRepository extends JpaRepository<User, String> {
        Optional<User> findByEmail(String email);
    }
    ```
*   **Abstracción Documental (MongoDB):** Interfaces que extienden de `MongoRepository<T, ID>` para interactuar de forma transparente con colecciones NoSQL sin necesidad de escribir código conductor manual. Ejemplos de uso incluyen:
    ```java
    public interface CourseRepository extends MongoRepository<Course, String> {
        List<Course> findByInstructorId(String instructorId);
    }
    ```

## 4. Desarrollo Guiado por Pruebas (TDD) y Estrategia de Calidad

En concordancia con la rama de pruebas (`pruebas`), cada microservicio de negocio incorpora una estrategia de verificación automatizada diseñada para validar componentes de forma aislada e integrada. La suite completa consta de **216 casos de prueba estructurados (124 pruebas unitarias y 92 de integración)**.

*   **Pruebas Unitarias (Unit Testing):** Ubicadas en `src/test/java/.../unit/`. Utilizan **JUnit 5** y **Mockito** para probar la lógica de negocio pura en aislamiento. Las dependencias externas (como bases de datos o servicios remotos) son sustituidas por simulacros (*mocks*), lo que permite probar exhaustivamente excepciones y flujos alternativos a gran velocidad (ej. `UserServiceImplTest.java`).
*   **Pruebas de Integración (Integration Testing):** Ubicadas en `src/test/java/.../integration/`. Utilizan **MockMvc** y **Testcontainers**. Testcontainers levanta imágenes reales de **PostgreSQL**, **MongoDB** y **RabbitMQ** en contenedores Docker efímeros para la duración de la suite de pruebas. Esto asegura que la lógica transaccional, las restricciones físicas de la BD y la comunicación de eventos se verifiquen contra entornos idénticos al de producción (ej. `UserControllerIT.java`, `UserRepositoryIT.java`).
*   **Firma de Cobertura de Código (JaCoCo):** Cada microservicio posee un validador automático JaCoCo en su archivo de configuración Maven (`pom.xml`) que detiene la fase de compilación si las pruebas unitarias y de integración no cubren al menos el **80% de las líneas de código** y el **80% de las ramas de decisión lógica**:
    ```xml
    <limit>
        <counter>LINE</counter>
        <value>COVEREDRATIO</value>
        <minimum>0.80</minimum>
    </limit>
    ```

## 5. Principios SOLID

El diseño de clases de la plataforma se rige por los principios **SOLID** para garantizar que los componentes sean altamente modulares y de bajo acoplamiento:

1.  **Single Responsibility Principle (SRP):** Cada clase tiene una única e inequívoca responsabilidad. Un controlador solo enruta peticiones; un servicio ejecuta lógica de negocio; un repositorio realiza operaciones de BD; un DTO transporta datos; y un mapper transforma tipos.
2.  **Open-Closed Principle (OCP):** Los componentes están abiertos a la extensión pero cerrados a la modificación. Esto se implementa utilizando interfaces de servicio (como `UserService` o `StorageService`). Por ejemplo, la carga de firmas digitales de EPP en `StorageServiceImpl` utiliza la interfaz `StorageService` para integrarse con AWS S3; si se requiriera migrar a Azure Blob Storage, solo se crearía una nueva implementación `AzureStorageServiceImpl` sin modificar ningún controlador o clase cliente.
3.  **Liskov Substitution Principle (LSP):** Las clases derivadas o implementadoras pueden sustituir de manera transparente a sus abstracciones. La implementación `UserServiceImpl` cumple rigurosamente el contrato definido en `UserService`, garantizando que cualquier controlador pueda consumirla sin conocer los detalles de su implementación.
4.  **Interface Segregation Principle (ISP):** Se favorece el uso de interfaces pequeñas y altamente específicas en lugar de interfaces masivas con múltiples responsabilidades. Por ejemplo, `QrService` expone únicamente el método `generateQrCode`, evitando mezclar la lógica de generación de códigos visuales con la gestión general de usuarios o almacenamiento.
5.  **Dependency Inversion Principle (DIP):** Los módulos de alto nivel no dependen de módulos de bajo nivel, sino de abstracciones. El framework de Spring gestiona la Inyección de Dependencias (DI) de forma automática. Se emplea el patrón de **inyección por constructor** (generado limpiamente mediante `@RequiredArgsConstructor` de Lombok), evitando el uso de acoplamiento rígido con la palabra clave `new`.
