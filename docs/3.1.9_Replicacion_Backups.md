# 3.1.9. Gestión de Replicación y Copias de Seguridad (Backups) en la Nube

Esta sección describe y sustenta la estrategia de **Alta Disponibilidad (HA)**, **Replicación de Datos** y **Copias de Seguridad (Backups)** implementada para las bases de datos de la plataforma **SafeIndustrial** utilizando infraestructura administrada en la nube (como AWS RDS para PostgreSQL y MongoDB Atlas para MongoDB).

La migración de configuraciones locales de replicación (archivos `master.conf` y `slave.conf` tradicionales) hacia servicios administrados en la nube permite delegar la gestión física de la infraestructura, garantizando acuerdos de nivel de servicio (SLA) de persistencia superiores al 99.99%.

A continuación se presenta el diagrama de arquitectura para la gestión de réplicas y respaldos en la nube, seguido de su justificación y fundamentación técnica.

### Sustentación de la Estrategia de Replicación y Backups

*   **Gestión de Replicación (Alta Disponibilidad y Rendimiento):**
    Para asegurar que la plataforma de seguridad industrial nunca interrumpa su monitoreo, se aplica un esquema de replicación híbrido:
    *   *Replicación Síncrona (Multi-AZ):* Las escrituras realizadas en la instancia primara (Zona A) se replican en tiempo real en una instancia espejo Standby (Zona B). Si la base de datos principal sufre una caída física o corrupción de datos, el servicio en la nube ejecuta un **Failover automático (conmutación por error)** en menos de 60 segundos, redirigiendo el tráfico del backend al nodo Standby sin pérdida de información (RPO = 0, RTO < 60s).
    *   *Replicación Asíncrona (Réplicas de Lectura):* Para evitar que las consultas complejas y la generación de reportes de KPIs degraden el rendimiento de las operaciones transaccionales críticas, se mantiene una réplica de lectura asíncrona (Zona C). El microservicio de analítica e informes lee directamente de este nodo, optimizando la latencia del sistema.
    *   *Replica Sets (MongoDB Atlas):* Para la base de datos documental (cursos y chats), se utiliza un clúster de tres nodos (un Primario y dos Secundarios) distribuidos geográficamente que gestionan la auto-recuperación ante caídas físicas mediante algoritmos de consenso automático.
*   **Gestión de Copias de Seguridad (Backups y Recuperación ante Desastres):**
    La estrategia de respaldos asegura la capacidad de restaurar el sistema ante errores humanos o fallos de software críticos mediante dos mecanismos clave almacenados de forma redundante en un almacén de objetos aislado (**AWS S3**):
    *   *Instantáneas Diarias Automáticas (Snapshots):* Cada madrugada se realiza un respaldo incremental completo de las bases de datos PostgreSQL y MongoDB con una retención de 30 días.
    *   *Recuperación en el Tiempo (Point-in-Time Recovery - PITR):* Mediante el archivado continuo de logs de transacciones (archivos **WAL** en PostgreSQL y **Oplog** en MongoDB), es posible restaurar las bases de datos a cualquier segundo específico de los últimos 14 días. Esto permite deshacer corrupciones de datos causadas por despliegues fallidos o ataques cibernéticos de forma precisa.
